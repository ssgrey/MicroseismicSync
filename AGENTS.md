# Repository Guidelines

## Project Structure & Module Organization
The solution root contains `MicroseismicSync.sln` and one WPF project in `MicroseismicSync/`. Core UI files live at the project root: `App.xaml`, `App.xaml.cs`, `MainWindow.xaml`, and `MainWindow.xaml.cs`. Assembly metadata, resources, and app settings are under `MicroseismicSync/Properties/`. Build outputs are generated in `MicroseismicSync/bin/` and `MicroseismicSync/obj/`; do not edit generated files there by hand.

## Build, Test, and Development Commands
Use Visual Studio 2022 or a Developer PowerShell with .NET Framework build tools installed.

- `msbuild MicroseismicSync.sln /p:Configuration=Debug`
  Builds the solution for local development.
- `msbuild MicroseismicSync.sln /p:Configuration=Release`
  Produces an optimized release build.
- `devenv MicroseismicSync.sln`
  Opens the solution in Visual Studio for XAML design, debugging, and resource editing.

`dotnet build` is not the preferred path for this legacy `.NET Framework 4.8` WPF project.

## Coding Style & Naming Conventions
Use 4-space indentation in C# and XAML. Keep public types, methods, and WPF elements in `PascalCase`; use `camelCase` for locals and private fields. Match code-behind filenames to their XAML views (`MainWindow.xaml` and `MainWindow.xaml.cs`). Prefer small event handlers and move reusable logic out of the UI layer as the app grows. Keep autogenerated designer/resource files untouched unless they are regenerated by Visual Studio.

## Testing Guidelines
There is no automated test project in the current solution. For new logic-heavy features, add a separate test project at the solution root, for example `MicroseismicSync.Tests/`, and name tests `MethodName_State_ExpectedResult`. Until a test suite exists, verify changes by building the solution and exercising the affected WPF workflow manually.

## Commit & Pull Request Guidelines
Git history is not available in this checkout, so follow a simple convention: use short, imperative commit subjects such as `Add sync status indicator` or `Fix startup config loading`. Keep commits focused. Pull requests should include a brief summary, list of manual verification steps, and screenshots for visible XAML/UI changes. Call out any `App.config` or `Properties/Settings.settings` changes explicitly.

## Configuration & Security Tips
Store environment-specific values in `App.config` or `Properties/Settings.settings`; do not hardcode machine-specific paths, credentials, or service endpoints in code. Review config diffs carefully before merging.
